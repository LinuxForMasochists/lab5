1. Основи Git
1.1. Яка різниця між Git та GitHub?

Git — локальна розподілена система контролю версій (працює на твоєму комп’ютері).

GitHub — хостинг для Git-репозиторіїв + додаткові інструменти (PR, Issues, CI/CD).

1.2. Що означає “розподілена” система контролю версій?

Кожен розробник має повну копію репозиторію з історією, а не лише частину.
Тому Git працює швидко й автономно — без доступу до сервера.

1.3. Яка різниця між git init та git clone?

git init — створює порожній локальний репозиторій у папці.

git clone — копіює вже існуючий віддалений репозиторій.
2. Staging Area та Commits
2.1. Що таке Staging Area і навіщо вона потрібна?

Це “проміжна зона”, де ти обираєш, які зміни увійдуть у наступний коміт.
Дозволяє підготувати коміт частинами.

2.2. Яка різниця між git add . та git add -A?

git add . — додає нові та змінені файли, але може пропустити видалені.

git add -A — додає всі зміни: нові, змінені та видалені (повна синхронізація).

2.3. Як написати гарний commit message?

Добре повідомлення має бути:

коротке (50–72 символи),

в наказовому тоні: "Fix bug", "Add login form",

з однією дією на коміт,

за потреби — додати опис у тілі коміта.

2.4. Як виправити останній коміт?
git commit --amend


Дозволяє змінити повідомлення або додати забуті файли.
3. Гілки та злиття
3.1. Що таке гілка в Git і чому вони “легковажні”?

Гілка — це просто вказівник на коміт.
Вона займає байти і створюється миттєво → тому вважається "легковажною".

3.2. Яка різниця між git merge та git rebase?

merge — створює новий коміт злиття, історія розгалужується.

rebase — переписує історію, ставить коміти “поверх” іншої гілки.

3.3. Що таке fast-forward merge?

Коли цільова гілка не має власних комітів, і вказівник просто пересувається вперед без створення merge-коміту.

3.4. Коли виникають merge конфлікти і як їх уникнути?

Конфлікти з’являються, коли один і той самий рядок файлу змінено в різних гілках.
Уникнення:

часті pull,

маленькі гілки,

чітке розділення задач.

3.5. Чому не можна робити rebase публічних комітів?

Тому що rebase переписує історію, і інші розробники отримають конфлікти та розсинхронізацію.
4. Віддалені репозиторії
4.1. Що таке origin?

origin — це ім’я за замовчуванням для основного віддаленого репозиторію.

4.2. Яка різниця між git fetch та git pull?

git fetch — забирає зміни з сервера, але не зливає їх.

git pull = fetch + merge/rebase.

4.3. Що робить git push --force і чому це небезпечно?

Перезаписує віддалену історію.
Можна видалити чужі коміти, якщо не обережно.

4.4. Як видалити віддалену гілку?
git push origin --delete branch_name
5. Скасування змін
5.1. Різниця між git reset --soft, --mixed, --hard

--soft — перемотає HEAD, але зміни залишаться в staging.

--mixed (за замовчуванням) — зміни залишаються в робочій директорії.

--hard — видалить всі зміни, повернувши стан до коміту.

5.2. Різниця між git reset та git revert

reset — змінює історію (небезпечний для публічних гілок).

revert — створює новий коміт, який відміняє попередній, історія зберігається.

5.3. Як скасувати git add?
git reset HEAD file

5.4. Як повернути видалений файл?

Якщо був у коміті:

git checkout HEAD -- path/to/file

6. Безпека
6.1. Що не повинно потрапляти в Git репозиторій?

паролі, токени,

секретні ключі,

node_modules, build-папки,

великі бінарні файли,

конфігурації з приватними даними.

6.2. Що робити, якщо випадково закомітили пароль?

Видалити з файлу.

Змінити пароль/токен (він уже скомпрометований).

Переписати історію:

git filter-repo або BFG Repo Cleaner

6.3. HTTPS vs SSH для GitHub — що краще?

SSH — зручніше та безпечніше для постійного користування.

HTTPS — підходить, якщо SSH недоступний або на корпоративних мережах.

6.4. Що таке Personal Access Token?

Це цифровий ключ, який замінює пароль GitHub для HTTPS-авторизації.
7. GitHub
7.1. Що таке Pull Request?

Запит на злиття гілки. Дозволяє провести рев'ю, тести та обговорення.

7.2. Яка різниця між Issues та Pull Requests?

Issues — задачі, баги, пропозиції.

Pull Requests — пропозиція змін коду.
Копія чужого репозиторію у твій GitHub-акаунт.
Використовується для внесення змін без доступу до оригіналу.

7.4. Що таке GitHub Actions?

CI/CD-платформа GitHub, яка автоматично запускає скрипти (тести, деплой, збірка).
8. Best Practices
8.1. Як часто треба робити коміти?

Часто, але логічно: одна задача = один коміт.

8.2. Скільки часу має жити feature branch?

Не більше кількох днів. Довгі гілки = більше конфліктів.

8.3. Як організувати git workflow в команді?

Популярні моделі:

GitFlow

GitHub Flow

Trunk-based Development

Головне: маленькі гілки, код-рев’ю, CI.

8.4. Чи треба комітити node_modules або інші залежності?

НІ.
Використовуй .gitignore, а залежності відновлюються через npm install або dotnet restore.
